# Contributing

## Commit Standards

- Use [Conventional Commits](https://www.conventionalcommits.org/).
- Write every commit message in English.
- Format: `type(scope): concise summary` or `type: concise summary`.
- Types: `feat`, `fix`, `docs`, `refactor`, `chore`, `test`, `build`, `ci`, `perf`.
- Breaking changes: add `!` after type/scope (e.g., `feat!: remove deprecated API`).

## Branch Naming

- Feature: `feature/{short-name}` (e.g., `feature/audio-badges`)
- Fix: `fix/{short-name}` (e.g., `fix/duplicate-insert`)
- Use lowercase kebab-case. Keep names short and descriptive.

## Project Knowledge Base

When `architecture.md` and `requirements.md` exist in the repository, all planning and implementation work **must** align with the documented patterns, conventions, and constraints. These files are generated by `/document` and represent the verified state of the project.

- Default location: `.docs/`, but they may reside anywhere in the repository.
- Skills that consume these files must search for them before proceeding.

## Workflow Updates

Before starting a new workflow cycle, check if the upstream repository has newer commits:

1. Compare the local workflow version against `https://github.com/This-Is-NPC/assisted_workflow` (use `gh api` or `git ls-remote`).
2. If updates exist, summarize what changed (new skills, modified skills, bug fixes, etc.).
3. Present the summary to the user and ask whether they want to update.
4. If the user confirms, run the appropriate install script with `--update` to apply changes while preserving `CONTRIBUTING.md`.
